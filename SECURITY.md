# Security Policy

## English
If you believe you have found a security vulnerability in JNDC, please avoid opening a public issue before a fix or mitigation is available.

Preferred reporting order:

* Use GitHub Private Vulnerability Reporting or a GitHub Security Advisory if it is enabled for this repository
* If private reporting is not available, contact the maintainer through the contact method exposed on the maintainer's GitHub profile
* Use a normal public issue only for non-sensitive hardening suggestions that do not expose a live exploit path

Please include:

* Affected version or commit
* Deployment context and prerequisites
* Reproduction steps or proof of concept
* Impact assessment
* Any suggested mitigation if available

Supported versions:

* The latest code on the default branch is the primary supported target for security fixes
* Older versions may receive no fix, delayed fixes, or a request to upgrade first

Disclosure expectations:

* Please allow the maintainer reasonable time to validate and mitigate the report before full public disclosure
* If a report turns out to be configuration misuse rather than a product vulnerability, the likely outcome will be documentation or hardening guidance instead of a code patch

## 中文
如果你认为自己发现了 JNDC 的安全漏洞，请在修复或缓解措施可用前，避免直接提交公开 issue。

建议报告顺序：

* 如果仓库启用了 GitHub 私密漏洞报告或 GitHub Security Advisory，优先使用该通道
* 如果没有私密报告通道，请通过维护者 GitHub 主页公开的联系方式私下联系维护者
* 只有在不涉及可利用细节的情况下，才使用公开 issue 提交一般性的安全加固建议

建议在报告中包含：

* 受影响版本或提交
* 部署场景和前置条件
* 复现步骤或 PoC
* 影响评估
* 如果已有缓解方案，也请一并说明

支持版本范围：

* 默认分支上的最新代码是安全修复的主要支持目标
* 旧版本可能不修复、延后修复，或需要先升级后再处理

公开披露预期：

* 请给维护者合理时间完成确认、缓解和修复，再进行完整公开披露
* 如果问题本质上是配置误用而不是产品漏洞，最终更可能给出文档或加固建议，而不是代码补丁
